Inference Economics

Why tokens are expensive, and why generating them costs more than reading them

By Abdullah Yousaf

Model providers charge more for the tokens a model generates than for the tokens you send it, usually several times more. A reasonable guess is that writing is harder than reading, or that the model is thinking about its answer. Neither is right.

A request runs in two phases. Prefill reads the prompt: the weights come out of memory once and get reused across all of the input tokens. Decode writes the reply one token at a time, and each new token requires another pass over the weights because tens of gigabytes of model weights cannot fit in the small, fast memory beside the compute units. The amount of compute per token is about the same in both, but the amount of data moved is not. Modern GPUs can perform the arithmetic far faster than they can read the weights from memory, so memory bandwidth, not compute, is what limits how quickly tokens can be generated.

In this write up, I follow a single token through both phases to show why generating tokens is more expensive than reading them, why the premium varies across providers, and what the major labs do to shrink it.

why this matters

The five biggest US cloud companies plan roughly $650 to $700 billion of capital spending in 2026, most of it AI compute.

Source: Futurum, Feb 2026.

At that scale, driving down the cost of tokens is what enables the following:

  • Agents can run longer. Agentic workflows are long-context loops where every step generates new tokens on top of a growing context. Cheaper output means more steps and tool calls within the same budget.
  • Reasoning becomes affordable. Long chains of reasoning are output-heavy, so lower token costs make it practical to spend more tokens thinking through a problem.
  • The strongest models become the default. As capable models get cheaper to serve, there is less pressure to route requests to weaker models to save money.
  • Large models fit on less hardware. Reducing the bytes moved per token means the same GPU can serve a larger model or more users.
  • Capable models can run locally. The same efficiency brings powerful models onto laptops, phones, edge devices, and IoT systems: protecting privacy, being offline, and having no per-token bill.

The same holds beyond Anthropic. Output is costlier everywhere, but the multiple varies: roughly 6 to 8x on the OpenAI and Google models, down near 2x on decode-optimized models like DeepSeek, with Anthropic's 5x sitting in between.

The same gap shows up on self-hosted open-source models once you count the GPU time each token costs so the cause is how the model interacts with the hardware. It comes down to one read versus many. In prefill, the weights are read from memory once and that read is shared across every input token. In decode the weights are read again for each output token, one at a time, and nothing is shared between tokens. Output moves a lot more data per token, which makes it bound by memory bandwidth, not compute. The rest of the explainer breaks down where that cost comes from.

02

Attention: every token reads every other token

When a model processes text, the meaning of each word depends on the words around it. Compare "I steered the boat toward the bank" with "I steered the client toward the bank". One word changes, and "bank" flips from the edge of a river to a financial one. The model cannot know in advance which earlier word decides the meaning, so it compares each token against every token before it. That comparison is what attention does.

Each token is projected into three vectors: a Query for what it is looking for, a Key for what it offers, and a Value for what it carries. Each token's Query is compared against every earlier Key to find which of them are most relevant. The higher the score, the more relevant that token is. A softmax then turns those relevance scores into weights that add up to 1, so more relevant tokens receive a larger share of the attention. Finally, the token's output is a weighted combination of the Value vectors with the most relevant tokens contributing the most.

softmax(QKT√d)V

The √d keeps the scores from becoming too large as the vectors get wider.

Because every Query touches every earlier Key, the number of comparisons grows with the square of the sequence length, O(n2). By doubling the length, we roughly quadruple the work.

03

The KV cache, and why it grows

Redoing all those comparisons at every step would waste work, so the model stores intermediate results. For each token it keeps two vectors, a Key and a Value, in a buffer called the KV cache. Every new token the model produces adds another entry.

the cache fills as the model writes
how big it gets (about a 7B model, fp16)

Each token is roughly 0.5 MB of cache.

context length4,096 tokens
KV cache sizeabout 2.0 GB

That is for one conversation. If we serve a hundred at once, memory fills with caches making less room for the weights to even fit.

The 0.5 MB per token cache is calculated as follows. Each token stores a Key and a Value in every layer and every attention head. For a 7B parameter model that is 2 (Key and Value) × 32 layers × 32 heads × 128 dimensions per head × 2 bytes in FP16, which comes to about 0.5 MB per token. From there the cache grows linearly with the conversation: a 2,000-token chat holds around 1 GB, and past 20,000 tokens it reaches 9.6 GB, close to the 14 GB of weights for the whole 7B model.

The general form is 2 · n_layers · n_kv_heads · d_head · bytes. In a standard model every query head carries its own Key and Value head, so n_kv_heads is 32 here, and that is the term Grouped-Query Attention and Multi-Query Attention address later by sharing Keys and Values across query heads to reduce the overall KV cache size.

That growth raises cost two ways. 1. The bigger the cache, the more data crosses the memory bus on every step: the model reads the whole cache out of HBM for each token it generates, so a 9.6 GB cache means 9.6 GB of KV traffic for a single token, on top of the 14 GB of weights. 2. And generation runs one token at a time, so a longer output repeats that read more times, each read pulling a bigger cache than the last. More bytes per read, and more reads, climbing together.

The expensive part is moving bytes out of HBM and not the arithmetic. The next sections show why that makes decode memory-bandwidth-bound and not compute-bound.

At 20,000 tokens of context the KV load alone is 9.6 GB, on top of 14 GB of weights, for each single token.

Each token reads all the weights plus the entire KV cache and uses them once. This is a matrix-vector multiply. Nothing is amortized, and the KV half grows every step, so the data moved per token only goes up.

The name for this gap is arithmetic intensity: floating-point operations performed per byte moved from memory. Prefill runs at high arithmetic intensity because a single weight load is reused across all N prompt tokens. Decode runs at an arithmetic intensity close to 1 because each weight read is only doing work for a single token before the weights have to be read again. A GPU can perform hundreds of operations for every byte it reads, so decode spends far more time waiting on memory than doing math. Prefill amortizes one weight read across the whole prompt. Decode repeats that read again on every token with no sharing between tokens, while also attending over a KV cache that grows with the conversation.

05

The memory-bandwidth wall

Section 04 showed that decode has to move for each token: every weight, plus the entire KV cache. If we turn that into a throughput limit, we get the ceiling on token generation.

This is the roofline model in practice. Every workload is limited by either compute or memory bandwidth and which one depends on its arithmetic intensity. Nvidia's B200 can perform around 2,250 trillion FP16 operations per second but reads memory at about 8 TB/s, which works out to around 280 operations per byte loaded. Decode is often closer to 1. For a 7B FP16 model, generating a token requires roughly 14 billion operations (about one multiply and one add per parameter) and reading roughly 14 GB of weights. The arithmetic finishes quickly but moving the data does not.

Most of the time is spent waiting for weights to arrive from HBM rather than performing computation. At that point, the constraint isn't compute. It's memory bandwidth: getting the weights to the compute units quickly enough.

tokens per second ≈ memory bandwidth ÷ bytes moved per token

As an approximation, the bytes moved per token are the model weights plus the current KV cache. The calculator lets you see how that changes as you vary the model, precision, GPU, and context length.

Back of the envelope. This counts weight reads plus KV cache traffic at batch one and ignores activations and kernel overhead. Real numbers differ, but the order of magnitude and the conclusion hold: decode speed is set by memory bandwidth, not compute. KV cache is counted at FP16; weight precision is set separately.

Three controls in the calculator change that limit, and each maps directly to a technique used in production:

  1. Precision. Fewer bytes per weight means fewer bytes to move. Dropping a 7B model from FP16 to int4 reduces the weights from 14 GB to 3.5 GB so the same GPU can generate more tokens per second.
  2. Batching. Read the weights once and reuse them across many users in a single pass. This is one of the biggest throughput levers in production because the weight read is the part that can be shared.
  3. Context length. The KV cache adds to the bytes moved per token and grows as the conversation does, so long contexts decode more slowly than short ones. This is the growing cache from section 04 now showing up as lost speed.

This is where the trade-off appears. Batching can share the weights, but it cannot share the KV cache because every user has their own. At a few thousand tokens, the weights dominate and batching can increase throughput several times over. If the context increases to hundreds of thousands, the KV cache starts to dominate instead and the same batch provides much less benefit.

This is evident, for example, in the calculator's default 8,192-token context, batching 32 users increases throughput from around 440 to 1,765 tokens per second. At 199,680 tokens, the same batch only moves it from about 70 to about 80. The cache helps the model remember the conversation but also limits how many users a GPU can serve.

06

Why input is cheap and output is expensive

The price gap follows directly from the two phases.

Input tokens go through prefill. They are read in parallel, and the weights come out of memory once for the entire prompt. That single read is split across every input token so each one is cheap.

Output tokens go through decode, one at a time. Nothing is shared and every new token reads the whole model and the entire cache out of memory again, and the cache is longer than it was the token before.

tokens in

Shared, parallel, cheap

1 read, shared by the whole prompt.

The weights come out of memory once during prefill and serve every input token at once. Each token's share of that read is small.

tokens out

Sequential, repeated, expensive

N reads, one per token generated (N is the length of the output).

Decode reads the full weights and the whole cache again for each token and the cache keeps growing. Nothing is shared so each token pays in full.

Input is read in bulk and that read is shared. It is compute-bound. Output is generated one token at a time with no reads being shared. It is memory-bandwidth-bound. That gap is why output tokens cost so much more than input tokens.

Everything after this is engineering to make those reads smaller and less frequent.

07

Techniques that reduce the cost

Once the bottleneck is clear, most inference work falls into a few groups. Each one does one of two things: 1. move fewer bytes per token, or 2. get more out of each read. Two are worth seeing in detail.

Continuous batching

raises throughput by keeping the GPU busy

Static batching groups requests together and waits for the slowest one to finish before starting the next batch. That leaves gaps where the GPU is not doing useful work. Continuous batching fills those gaps. As soon as one request finishes, a new one takes its place. The GPU stays busy and throughput goes up as the same weight read is shared across more users. This is one of the core ideas behind serving systems like vLLM, SGLang, TensorRT-LLM, and NVIDIA Dynamo.

Speculative decoding

spends idle compute to skip memory trips

Decode is memory-bound as the GPU spends much of its time waiting on memory rather than doing arithmetic. Speculative decoding uses that spare compute. A smaller draft model predicts several tokens ahead and the larger model then checks them all in a single parallel pass which looks much more like prefill than decode. Tokens the larger model agrees with are accepted together, so one pass over the weights can produce multiple output tokens. The first token it disagrees with becomes the new starting point.

The speedup depends on how often the draft model is right. The final output is unchanged because the larger model still decides which tokens are accepted.

03

KV cache management

shrinks the cache and wastes less memory

The KV cache grows with every token, so reducing it directly reduces the amount of data that has to be stored and moved. PagedAttention manages the KV cache much like an operating system manages memory: in small fixed-size pages rather than large contiguous blocks. That dramatically reduces fragmentation, making better use of GPU memory and allowing more users to share the same card. Grouped-Query Attention (GQA) and Multi-Query Attention (MQA) attack the problem from the model side. Instead of every query head having its own Keys and Values, multiple heads share them. The result is a much smaller KV cache and fewer bytes moved per token.

04

FlashAttention

cuts prefill memory traffic

A naive attention implementation stores the full n × n intermediate attention matrix in HBM, then reads it back again. As sequence length grows, both the memory storage requirements and memory traffic grow quadratically.

FlashAttention avoids building that matrix. Instead, it processes attention in tiles that fit in the GPU's fast on-chip shared memory (SMEM) without writing the full matrix to HBM. The output is the same and the difference is that far less data moves through memory. FlashAttention-2 increased parallelism and FlashAttention-3 was redesigned around Hopper's tensor cores and asynchronous execution. Together they enable the practicality of today's long-context models.

Note: I built one of these by hand in CUDA last quarter, so I am happy to go deeper on it live.

05

Mixture of experts

shrinks the weights read per token

A dense model reads all of its weights for every token. A mixture of experts splits the network into many sub-models and routes each token to only a few of them, so only a slice of the weights moves per token. The full model still has to fit in memory, but the per-token bandwidth drops to the active part. This technique is now standard for the frontier models.

Case study

DeepSeek V4

A real model demonstrates these techniques together. DeepSeek released the V4 family in April 2026 as open weights under an MIT license. V4-Pro has the strongest open-weights coding score, 80.6% on SWE-bench Verified, while charging $0.87 per million output tokens against Opus 4.8's $25. That is around 30x cheaper on the expensive side. It does this by working on both sides of the cost equation from section 06 at once.

Starting with the weights term, V4-Pro is a Mixture of Experts (MoE), the technique in the card above: 1.6 trillion parameters total, but a router sends each token to only a slice of them so just 49 billion are read per token. That puts its per-token bandwidth cost near a 49B model but still keeping the knowledge and capability of a larger one. The other ~1.55 trillion parameters sit in HBM and don't cross the memory bus for that token.

Then for the cache term: Earlier DeepSeek models already compressed the KV cache with Multi-head Latent Attention (MLA), storing one small shared vector per token instead of full keys and values. V4 goes further with sparse, compressed attention: a hybrid of Compressed Sparse Attention (CSA) and Heavily Compressed Attention (HCA) that squeezes every few tokens into a single entry and has each query attend to only the top-k of them. So at a million tokens of context, V4-Pro uses about 10% of the previous generation's KV cache. Thus, the KV cache term that dominates long conversations shrinks too.

It also runs mixed-precision quantization: FP4 (4-bit floating point) for the experts, FP8 (8-bit) for attention, normalization, and the router, with the FP4 on the experts doing most of the work. And it predicts more than one token per step, so a single pass through the weights generates several tokens instead of one.

So V4-Pro uses the same two ideas from this talk: move fewer bytes per token, and get more out of each read. By addressing both areas, it was able to undercut the frontier labs on price by a significant amount.
